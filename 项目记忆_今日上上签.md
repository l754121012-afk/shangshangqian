## 当前项目定位

项目名称：今日上上签

产品类型：微信小程序优先。

目标人群：急功近利、焦虑型、渴望即时反馈、喜欢轻玄学和每日仪式感的年轻用户。

核心定位：不是算命，而是“每日心理仪式感工具”。用抽卡、今日宜忌、情绪打卡、微行动和分享海报，把用户对不确定性的焦虑转化为当天可执行的小行动。

## 已确定的重要判断

1. 广告收入优先，不优先依赖版号、重付费或复杂会员体系。
2. 在最初调研的多个方案中，“AI 今日上上签”是第一优先级。
3. 产品名已从“今日指引”统一改为“今日上上签”，因为更讨喜，更符合目标人群的心理预期。
4. 合规边界很重要：不做八字命盘、真人大师、命运预测、医疗心理诊断、投资建议。
5. 产品表达应是“娱乐 + 自我反思 + 今日行动建议”，避免迷信承诺。

## MVP 建议

第一版只做最小闭环：

1. 今日抽卡：用户打开即可抽一张今日上上签。
2. 今日宜忌：每张卡有一个宜、一个忌。
3. 微行动：给用户一个 3-10 分钟内可以完成的小任务。
4. 分享海报：生成适合朋友圈、小红书、微信群传播的卡片图。
5. 激励视频再抽：看广告解锁补充卡。

情绪打卡和答案之书可以作为第二个版本加入。

## 技术方向

推荐技术路线：

1. 微信原生小程序。
2. 后端使用微信云开发 CloudBase 或轻量 Node.js API。
3. AI Key 只能放后端，不能暴露在小程序前端。
4. 内容生成采用“模板库 + AI 改写 + 缓存 + 安全过滤”。
5. 分享图使用小程序 Canvas 生成。
6. 广告优先接入 banner 和激励视频，插屏谨慎使用。


## 协作与版本规则

### 任务拆分

后续开发必须按强解耦原则拆分任务。每个模块都要有独立的输入、输出、数据结构和验收标准，尽量不依赖其他模块的内部实现。

模块之间只通过稳定接口协作，包括 API 字段、JSON Schema、组件 props、页面路径、埋点事件名和 mock 数据。不同 agent 可以分别处理不同模块，但不得直接修改其他模块的内部逻辑；确需跨模块调整时，必须先更新接口约定和项目记忆。

### 流水线协作

项目协作分为三个阶段：

| 阶段 | 职责 | 产出 |
| --- | --- | --- |
| 规划 | 定义目标、边界、接口、验收标准 | PRD、模块说明、数据结构 |
| 执行 | 按模块独立实现，不扩大范围 | 页面、组件、接口、模板、测试样例 |
| 审查 | 检查接口、体验、合规、性能、合并风险 | 审查记录、修复建议、版本判断 |

规划、执行与审查必须分离。执行阶段如果发现需求不清晰，先回到文档补充说明，不直接凭感觉扩大功能。

### 并行开发原则

整体设计要拆成多个相互独立的模块，使不同 agent 分别开发后也能顺利合并。推荐先冻结以下契约：

1. 页面路径。
2. 组件名称和组件输入输出。
3. API 路径、请求字段、响应字段。
4. 数据库集合和字段。
5. 内容模板 JSON 字段。
6. 埋点事件名。
7. 广告位名称和触发规则。

每个 agent 只处理自己负责的模块。涉及字段、路径、事件名、共享样式、广告规则、内容风控规则的变更，都必须先写入计划文档，再修改代码。

### Git 版本节点

每个关键节点通过后，都要向项目负责人询问：

```text
当前节点已完成并通过检查，是否上传一个版本至 Git，方便后续回滚和对比？
```

关键节点包括：

1. 项目骨架建立完成。
2. MVP 范围和接口契约定稿。
3. 首页、抽卡页、结果页主流程跑通。
4. 内容模板和 AI 生成兜底跑通。
5. 分享海报生成跑通。
6. 广告激励流程跑通。
7. 内测修复完成。
8. 提审前冻结版本。

Git 不只是最后上线前使用，而是每个可验证节点都要保存版本。远程上传或创建远程仓库前，需要确认仓库地址、分支名和是否允许推送。

## 第二版财运风 v2 增强层

已选择第二版财运风作为主要视觉方向。v2 优化不是替代原本的每日心理仪式感主流程，而是在主流程之后增加留存、分享和轻变现增强层。

主流程仍然是：

```text
每日抽卡 → 今日建议 → 微行动 → 分享 / 记录
```

增强层包括：

1. 稀有好运牌特效：SSR、福运、大吉等高稀有牌可以使用跳动火焰、金色光环、烫金扫光和特殊抽取动画。
2. 验证成功后的感谢入口：用户主动确认“今天真的有点准”后，才出现轻量支持或“添一束好运火”之类的隐晦收益入口。
3. 好友比手气：允许用户分享卡片并与好友比较今日手气、稀有度和好运值，但不能制造输赢压力。
4. 群体好运火种池：多位好友的好运聚在一起后，生成一句正向鼓励，适合微信群传播。
5. 牌册与稀有度：展示以往抽到的牌，按稀有度和牌组组织，让用户有收集动机。集齐牌组后输出阶段性自我观察，不输出命运结论。
6. 广告转运：抽到低运牌时，可以通过激励视频获得更温和、更可行动的转机卡。表达为“换个角度”，不能暗示看广告真实改变运气。
7. 负面牌消解：负面牌不做稀有收藏。用户输入“我已克服 / 我已解决”等完成语后，卡片合并消失，只留下行动记录。

### v2 边界规则

1. 稀有牌、牌册、好友比较是增强模块，不改变 MVP 主闭环。
2. 商业化入口只能在用户主动确认正向验证后出现，不能在负面情绪、低运牌、焦虑语境中出现。
3. 所有“转运”表达都要解释为“换一种行动角度”，不能表达成现实运气被改变。
4. 好友比较只比较今日手气和卡牌稀有度，不做羞辱性排名，不提示“你今天比别人差”。
5. 群体好运只输出正向鼓励，不绑定真实财富承诺。
6. 负面牌不应强化收藏价值，避免用户反复查看负面内容。

这部分对盈利和留存有正向作用，但必须保持“娱乐 + 自我反思 + 今日行动建议”的产品核心。

## 抽签次数与盈利规则

当前采用“每日免费 3 抽 + 广告解锁额外体验”的规则。

推荐节奏：

1. 每日免费 3 抽。
2. 第 1 抽为今日主签，权重最高，写入今日记录。
3. 第 2-3 抽为补充签，用来给用户换一个角度，不覆盖主签。
4. 第 4 抽开始需要观看激励视频或模拟广告。
5. 抽到低运或负面牌时，可额外提供 1 次广告转运。
6. 补充签每天建议 1 次，避免内容和稀有度被过快消耗。
7. 用户主动验证“今天真的有点准”后，可出现轻量感谢入口，但它不是主收入。

收入优先级：

```text
激励视频再抽 / 补充签 / 转运 > 分享拉新带来的广告展示 > 牌册留存 > 验证成功后的感谢入口
```

边界：

1. 免费次数不能无限，否则转运、补充签和牌册收集都会失去支撑。
2. 广告解锁表达为“补充角度”“转机卡”“再看一张”，不能表达为真实改变命运。
3. 感谢入口只能在用户主动验证成功且情绪状态正向时出现。

## 牌册差异化展示方案（v17 已确认）

经多轮效果图迭代确认，牌册展示采用以下方案：

### 4种签等级与视觉规则

| 签等级 | 底色 | 边框 | 文字色 | 印章 | 底纹 | 特效 |
|--------|------|------|--------|------|------|------|
| 大吉签 | 亮红渐变(#e84a2a→#a01808) | 2.5px金色(#f0d0a0) | 金色(#f0d0a0) | 深红底白字 | 柱状火焰+十字星光(清晰几何，opacity 0.08) | 扫光+旋转光晕+呼吸发光 |
| 上上签 | 暖米色渐变(#faf0d0→#f0e0b0) | 2px金色(#e0c880) | 深棕(#4a2818) | 深红底白字 | 流畅圆弧(border-top, opacity 0.1) | 无 |
| 上签 | 浅暖棕金(#f5e8d0→#dcc090) | 2px半透明棕 | 深棕(#5a4030) | 棕红底白字 | 无 | 无 |
| 平签 | 深暖棕金(#c8a060→#8a6838) | 1.5px半透明棕 | 奶白(#fff8f0) | 半透明深红 | 无 | 无 |

### 暖色传递规律
米色(上上签) → 浅暖棕金(上签) → 深暖棕金(平签)，颜色从浅到深自然过渡。

### 共性规则
- 所有签等大小网格展示，不区分大小
- 每张签有下段渐变装饰
- 每张签有内边框装饰线（::before伪元素）
- 右上角深红印章
- 左上角等级标签
- 中间：签名(大字) + 签文(小字)
- 底部：微行动建议

### 底纹设计原则
- 只有大吉签和上上签有底纹
- 底纹全部使用大尺度清晰几何形状（水平/垂直矩形、圆弧），无斜边三角形、无碎块椭圆
- opacity极低(0.08-0.1)，z-index在文字之下(z-index:1)
- 不影响任何文字和标志的辨识度

### 大吉签专属特效
- 扫光：金色光带从左到右循环扫过，3.5秒周期，z-index:1
- 旋转光晕：两个虚化光斑绕中心旋转，6秒周期，z-index:0
- 呼吸发光：box-shadow脉动，3秒周期

## 漂流签社交功能方案（v17 已确认）

### 核心流程
用户可把自己抽到的签装进漂流瓶，匿名或实名送给陌生人或好友，附上一句话感言。收到漂流签的用户可查看感言，选择感谢、回赠、或加为好友。

### 功能细节
1. 漂流签列表：展示收到的签（匿名/实名+地区），每条含发送者头像、名字、签类型标签、感言、来源信息
2. 操作按钮：感谢 / 回赠一张签 / 加为好友（实名签出现加好友按钮）
3. 好友申请弹窗：收到漂流签后点击"加为好友"弹出，显示发送者信息+附言+同意/拒绝
4. 送出入口：底部固定栏"送出一张好运签"
5. 标签切换：收到的签 / 我送出的
6. 平台承托语：不绑定真实财富承诺，只输出正向鼓励

### 合规边界
- 漂流签只传递正能量和鼓励，不传播负面情绪
- 好友申请是双向选择，不强制
- 不展示发送者真实个人信息（除非实名选择）

## v22 版本设计决策（2026-06-28）

### 签卡体系（20张）

| 等级 | 数量 | 概率 |
|------|------|------|
| 大吉签 | 4张 | 20% |
| 上上签 | 5张 | 20% |
| 上签 | 4张 | 20% |
| 平签 | 4张 | 20% |
| 解签 | 3张 | 20% |

每种等级概率均衡，解签（负面签）从0.8%提升到20%，确保用户能体验到"我已克服"消解功能。

### 界面优化

1. **去掉SSR/SR等游戏化标签**：抽签结果页和牌册统一使用"大吉签/上上签/上签/平签/解签"中文等级，不再显示SSR/SR/R/N/NEG。
2. **tab栏5个一排**：签/测/漂/火/我五个tab在底部单排展示，主文字15px，副标题8px。
3. **抽签结果页特效**：大吉签抽出时统一使用扫光+旋转光晕+呼吸发光（和牌册一致），去掉之前5个橙色圆球"痘痘"特效。

### 牌册展示

1. **按等级分类排列**：不再按收集时间排序，改为大吉签/上上签/上签/平签分组，每组显示"等级名 x/总数"标题。
2. **未收集签签背面效果**：保留对应等级的边框和背景色，整体opacity 0.35 + 遮罩层覆盖内容，只显示"?"和"尚未收集"。
3. **分组标题只占一行**：使用`grid-column: 1 / -1`横跨两列，紧凑margin，不占完整签位。

### 交互功能

1. **"我已克服"输入消除**：抽到负面签后，点击按钮弹出输入框，用户输入文字（如"我已克服"）后确认消除，toast显示用户输入内容。
2. **打赏功能**："添一束好运火"改为打赏弹窗，可选¥1/¥3(推荐)/¥5，非广告收益，不涉及概率无需版号。
3. **好友比手气**：好运值改为当日累计总和（所有抽卡分数相加），显示"今日累计好运值 X 分（已抽X张）"，好友排名动态更新。
4. **用户反馈问卷**："我"页面添加4+1题问卷（第5题自由留言，50字以内，可选），完成奖励1次免费抽签。
