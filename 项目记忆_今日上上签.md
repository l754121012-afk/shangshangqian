## 当前项目定位

项目名称：今日上上签

产品类型：微信小程序优先。

目标人群：急功近利、焦虑型、渴望即时反馈、喜欢轻玄学和每日仪式感的年轻用户。

核心定位：不是算命，而是“每日心理仪式感工具”。用抽卡、今日宜忌、情绪打卡、微行动和分享海报，把用户对不确定性的焦虑转化为当天可执行的小行动。

## 已确定的重要判断

1. 广告收入优先，不优先依赖版号、重付费或复杂会员体系。
2. 在最初调研的多个方案中，“AI 今日上上签”是第一优先级。
3. 产品名已从“今日指引”统一改为“今日上上签”，因为更讨喜，更符合目标人群的心理预期。
4. 合规边界很重要：不做八字命盘、真人大师、命运预测、医疗心理诊断、投资建议。
5. 产品表达应是“娱乐 + 自我反思 + 今日行动建议”，避免迷信承诺。

## MVP 建议

第一版只做最小闭环：

1. 今日抽卡：用户打开即可抽一张今日上上签。
2. 今日宜忌：每张卡有一个宜、一个忌。
3. 微行动：给用户一个 3-10 分钟内可以完成的小任务。
4. 分享海报：生成适合朋友圈、小红书、微信群传播的卡片图。
5. 激励视频再抽：看广告解锁补充卡。

情绪打卡和答案之书可以作为第二个版本加入。

## 技术方向

推荐技术路线：

1. 微信原生小程序。
2. 后端使用微信云开发 CloudBase 或轻量 Node.js API。
3. AI Key 只能放后端，不能暴露在小程序前端。
4. 内容生成采用“模板库 + AI 改写 + 缓存 + 安全过滤”。
5. 分享图使用小程序 Canvas 生成。
6. 广告优先接入 banner 和激励视频，插屏谨慎使用。


## 协作与版本规则

### 任务拆分

后续开发必须按强解耦原则拆分任务。每个模块都要有独立的输入、输出、数据结构和验收标准，尽量不依赖其他模块的内部实现。

模块之间只通过稳定接口协作，包括 API 字段、JSON Schema、组件 props、页面路径、埋点事件名和 mock 数据。不同 agent 可以分别处理不同模块，但不得直接修改其他模块的内部逻辑；确需跨模块调整时，必须先更新接口约定和项目记忆。

### 流水线协作

项目协作分为三个阶段：

| 阶段 | 职责 | 产出 |
| --- | --- | --- |
| 规划 | 定义目标、边界、接口、验收标准 | PRD、模块说明、数据结构 |
| 执行 | 按模块独立实现，不扩大范围 | 页面、组件、接口、模板、测试样例 |
| 审查 | 检查接口、体验、合规、性能、合并风险 | 审查记录、修复建议、版本判断 |

规划、执行与审查必须分离。执行阶段如果发现需求不清晰，先回到文档补充说明，不直接凭感觉扩大功能。

### 并行开发原则

整体设计要拆成多个相互独立的模块，使不同 agent 分别开发后也能顺利合并。推荐先冻结以下契约：

1. 页面路径。
2. 组件名称和组件输入输出。
3. API 路径、请求字段、响应字段。
4. 数据库集合和字段。
5. 内容模板 JSON 字段。
6. 埋点事件名。
7. 广告位名称和触发规则。

每个 agent 只处理自己负责的模块。涉及字段、路径、事件名、共享样式、广告规则、内容风控规则的变更，都必须先写入计划文档，再修改代码。

### Git 版本节点

每个关键节点通过后，都要向项目负责人询问：

```text
当前节点已完成并通过检查，是否上传一个版本至 Git，方便后续回滚和对比？
```

关键节点包括：

1. 项目骨架建立完成。
2. MVP 范围和接口契约定稿。
3. 首页、抽卡页、结果页主流程跑通。
4. 内容模板和 AI 生成兜底跑通。
5. 分享海报生成跑通。
6. 广告激励流程跑通。
7. 内测修复完成。
8. 提审前冻结版本。

Git 不只是最后上线前使用，而是每个可验证节点都要保存版本。远程上传或创建远程仓库前，需要确认仓库地址、分支名和是否允许推送。

## 内容策略

每张签卡建议包含：

1. 关注主题：事业、学习、感情、金钱、状态、人际。
2. 签名：例如“稳住签”“轻装签”“回神签”。
3. 今日关键词。
4. 一句情绪承接。
5. 今日建议。
6. 今日宜。
7. 今日忌。
8. 3 分钟微行动。
9. 适合分享的短句。

## 运营方向

主要渠道：

1. 小红书：指引卡截图、情绪共鸣文案。
2. 抖音：每日抽卡短视频和评论互动。
3. 微信群：每日打卡和分享裂变。
4. 朋友圈：海报传播。

关键窗口：

1. 2026 年 8 月：暑期冷启动。
2. 2026 年 9 月：开学季放量。
3. 2026 年 12 月：年末专题预热。
4. 2027 年 1 月：新年签和新年目标爆发。
